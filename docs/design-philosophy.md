# 設計方針・思想

## なぜ TypeSpec か

OpenAPI (Swagger) を直接書くのではなく TypeSpec を採用した理由:

- **OpenAPI JSON/YAML は人間が書くものではない**: OpenAPI 仕様を直接記述するのは冗長で読みづらく、レビューも困難。TypeSpec は簡潔な DSL で同等のスキーマを定義でき、可読性・保守性が高い
- **複数サービス間でのモデル共有**: `typespec/shared/model.tsp` で共通モデルを定義し、各サービス（admin, game-server）が import して拡張できる。OpenAPI 単体ではこの仕組みが弱い
- **Microsoft 製で ASP.NET との親和性が高い**: TypeSpec は Microsoft が開発しており、.NET エコシステムとの統合が期待できる。C# サーバーエミッターも公式で開発中
- **言語としての表現力**: デコレータ（`@route`, `@path`, `@query` 等）による直感的な API 定義、テンプレート・ユニオン型・enum 等の型システムが充実している

### 他の選択肢との比較

| ツール | 不採用の理由 |
|---|---|
| OpenAPI 直接記述 | 冗長すぎてメンテナンスが困難 |
| Swagger Editor | GUI は便利だがバージョン管理・レビューに不向き |
| Protocol Buffers (gRPC) | REST API との互換性がない。Web フロントエンドとの相性が悪い |
| GraphQL | 管理ツールの CRUD には過剰。OpenAPI エコシステム（NSwag, API Platform Admin）を活用できない |

## なぜレイヤードアーキテクチャか

各バックエンドサービスを Presentation / Application / Infrastructure の 3 プロジェクトに分割している理由:

- **NSwag 生成コードとビジネスロジックの分離**: NSwag が生成する抽象コントローラーと DTO は Presentation 層に閉じ込め、ドメインロジックは Application 層で独立して管理する。生成コードの再生成でビジネスロジックが壊れない
- **`internal` によるアクセス制御**: C# の `internal` 修飾子とプロジェクト分割により、レイヤー間の依存方向をコンパイル時に強制できる。Infrastructure の Entity や Repository 実装が Presentation から直接参照されることを防ぐ
- **テスタビリティ**: Application 層が純粋な C# classlib（フレームワーク依存なし）なので、ドメインモデルやアプリケーションサービスの単体テストが容易
- **概念実証としてのシンプルさ**: クリーンアーキテクチャやヘキサゴナルアーキテクチャほど複雑にせず、3 層で十分な関心の分離を実現できる。学習コストとのバランスが良い

### なぜクリーンアーキテクチャではないか

- 本プロジェクトは概念実証・サンプルであり、過度な抽象化は学習の妨げになる
- 3 層で「生成コード」「ビジネスロジック」「永続化」の分離は十分に達成できている
- 必要に応じてクリーンアーキテクチャに発展させることは可能（現在の構造はその前段階として機能する）

## スキーマ駆動開発とコントラクトファースト開発に関する所感

### 良い点

- **API 仕様の実装強制**: コードの自動生成によって API 仕様の実装を強制できるため、API 仕様と実装の乖離を防ぐ仕組みを作りやすい
- **ドキュメントの陳腐化防止**: API 仕様そのものがドキュメントになるため、ドキュメントの陳腐化問題にも対処しやすい
- **分業のしやすさ**: API 仕様で定義したインターフェースを契約としてこれを前提に開発を行えば良いので、フロントエンド・バックエンド間の分業が行いやすい
- **API の一覧性**: 全 API の定義が TypeSpec に集約されるため、一覧性が高い

### 気になる点

- **コード生成の柔軟性**: コード生成の柔軟性は言語/フレームワークとそのコードジェネレーターに依存する。エコシステムやコミュニティの充実度にも影響を受ける

## 「共有」の正しいレイヤー — スキーマ共有 vs コード共有

### 問い

TypeSpec の `shared/model.tsp` にモデルを定義して複数サービスで共有しているが、API クライアントや DTO の C# コードも共有ライブラリ（クラスライブラリ）として使い回すべきか？ 例えば、game server の API クライアントをクラスライブラリにしておけば、admin.backend や将来の Unity プロジェクトで共用できるのでは？

### 回答：スキーマを共有し、コードは各自生成する

スキーマ駆動開発における「共有」は **スキーマ（定義）レベルで共有し、コード（実装）は生成で各自持つ** のが正しいアプローチ。

```
typespec/game-server/
  → openapi.gameserver.json                  ← 共有されるのはここ
    → NSwag (C# server controllers)          ← game.server 用
    → NSwag (C# HttpClient)                  ← admin.backend 用
    → OpenAPI Generator (C# Unity 用)        ← Unity プロジェクト用（将来）
```

**「同じコードを書いている」のではなく「同じスキーマから生成している」だけなら、それは重複ではなく正しい分離。**

### なぜ API クライアントの共有ライブラリは推奨しないか

#### プラットフォーム間で HTTP 基盤が異なる

| | ASP.NET Core (admin.backend) | Unity |
|---|---|---|
| HTTP | `HttpClient` + `IHttpClientFactory` | `UnityWebRequest` |
| JSON | `System.Text.Json` | `JsonUtility` or Newtonsoft |
| DI | `Microsoft.Extensions.DependencyInjection` | なし（自前 or VContainer 等） |
| async | `Task` + `CancellationToken` | `Coroutine` or `UniTask` |
| .NET | net10.0 | .NET Standard 2.1 / Unity 独自 |

共有ライブラリにすると、最も制約が厳しいプラットフォーム（Unity）に引きずられて、ASP.NET Core 側で `IHttpClientFactory` やリトライポリシー（Polly）等のエコシステムが使えなくなる。

#### 同じ .NET サービスが複数あっても

2 サービス程度なら各自で NSwag クライアントを生成する方が良い:

- NSwag で生成するクライアントはほぼコピペ相当で、手書きコードの重複ではない
- 各サービスが必要なエンドポイントだけ使える（admin は `GET /players` のみ、別サービスは `POST /players` も等）
- デプロイ独立性が保たれる（共有ライブラリは NuGet パッケージ管理が必要になり、ビルド・リリースの結合度が上がる）

### 共有ライブラリ検討の判断基準

```
.NET サービス 2 つ → 各自で NSwag クライアント生成
.NET サービス 3 つ以上 + 共通の横断的関心事 → 共有ライブラリ検討
Unity 等 .NET 外 → 常に個別生成
```

「共通の横断的関心事」の具体例:
- 認証ヘッダーの付与ロジック
- リトライ / サーキットブレーカーのポリシー
- 共通のエラーハンドリング・ログ出力

## C# 共有ライブラリの適切な使いどころ

上記を踏まえ、C# の共有ライブラリ（クラスライブラリ）が正当化されるのは **「同じランタイム上で、同じ実装ロジックを複数箇所が使う」** とき。

### ✅ 使うべき場面

| カテゴリ | 例 |
|---|---|
| **ドメインロジックの共有** | 複数サービスが同じ計算ロジックを持つ（課金計算、ダメージ計算式など） |
| **横断的関心事** | 認証・認可ヘルパー、カスタムログフォーマッタ、共通ミドルウェア |
| **社内フレームワーク** | 独自の Result 型、エラーハンドリング規約、ページネーション基盤 |
| **テストユーティリティ** | カスタムアサーション、テストデータビルダー、モックヘルパー |

### ❌ 使わない方がいい場面

| カテゴリ | 理由 |
|---|---|
| **DTO / API 契約の共有** | スキーマ生成（TypeSpec → OpenAPI → NSwag）で解決。ライブラリにするとサーバーとクライアントが密結合する |
| **薄いラッパー** | `StringUtils.IsNullOrEmpty()` 程度なら各自で書いた方が依存が減る |
| **異なるランタイム間** | ASP.NET Core ↔ Unity のように制約が違うプラットフォーム |

### 判断フローチャート

```
共有したいものは何？
  ├─ API の型定義 → TypeSpec + コード生成
  ├─ ビジネスロジック → 共有ライブラリ ✅
  ├─ インフラ基盤（認証・リトライ等） → 共有ライブラリ ✅（同一ランタイムなら）
  └─ HTTP クライアント → 各サービスで生成（プラットフォーム差があるなら確実に）
```
