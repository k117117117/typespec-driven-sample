# 設計方針・思想

## なぜ TypeSpec か

OpenAPI (Swagger) を直接書くのではなく TypeSpec を採用した理由:

- **OpenAPI JSON/YAML は人間が書くものではない**: OpenAPI 仕様を直接記述するのは冗長で読みづらく、レビューも困難。TypeSpec は簡潔な DSL で同等のスキーマを定義でき、可読性・保守性が高い
- **複数サービス間でのモデル共有**: `typespec/shared/model.tsp` で共通モデルを定義し、各サービス（admin, game-server）が import して拡張できる。OpenAPI 単体ではこの仕組みが弱い
- **Microsoft 製で ASP.NET との親和性が高い**: TypeSpec は Microsoft が開発しており、.NET エコシステムとの統合が期待できる。C# サーバーエミッターも公式で開発中
- **言語としての表現力**: デコレータ（`@route`, `@path`, `@query` 等）による直感的な API 定義、テンプレート・ユニオン型・enum 等の型システムが充実している

### 他の選択肢との比較

| ツール | 不採用の理由 |
|---|---|
| OpenAPI 直接記述 | 冗長すぎてメンテナンスが困難 |
| Swagger Editor | GUI は便利だがバージョン管理・レビューに不向き |
| Protocol Buffers (gRPC) | REST API との互換性がない。Web フロントエンドとの相性が悪い |
| GraphQL | 管理ツールの CRUD には過剰。OpenAPI エコシステム（NSwag, API Platform Admin）を活用できない |

## なぜレイヤードアーキテクチャか

各バックエンドサービスを Presentation / Application / Infrastructure の 3 プロジェクトに分割している理由:

- **NSwag 生成コードとビジネスロジックの分離**: NSwag が生成する抽象コントローラーと DTO は Presentation 層に閉じ込め、ドメインロジックは Application 層で独立して管理する。生成コードの再生成でビジネスロジックが壊れない
- **`internal` によるアクセス制御**: C# の `internal` 修飾子とプロジェクト分割により、レイヤー間の依存方向をコンパイル時に強制できる。Infrastructure の Entity や Repository 実装が Presentation から直接参照されることを防ぐ
- **テスタビリティ**: Application 層が純粋な C# classlib（フレームワーク依存なし）なので、ドメインモデルやアプリケーションサービスの単体テストが容易
- **概念実証としてのシンプルさ**: クリーンアーキテクチャやヘキサゴナルアーキテクチャほど複雑にせず、3 層で十分な関心の分離を実現できる。学習コストとのバランスが良い

### なぜクリーンアーキテクチャではないか

- 本プロジェクトは概念実証・サンプルであり、過度な抽象化は学習の妨げになる
- 3 層で「生成コード」「ビジネスロジック」「永続化」の分離は十分に達成できている
- 必要に応じてクリーンアーキテクチャに発展させることは可能（現在の構造はその前段階として機能する）

## スキーマ駆動開発とコントラクトファースト開発に関する所感

### 良い点

- **API 仕様の実装強制**: コードの自動生成によって API 仕様の実装を強制できるため、API 仕様と実装の乖離を防ぐ仕組みを作りやすい
- **ドキュメントの陳腐化防止**: API 仕様そのものがドキュメントになるため、ドキュメントの陳腐化問題にも対処しやすい
- **分業のしやすさ**: API 仕様で定義したインターフェースを契約としてこれを前提に開発を行えば良いので、フロントエンド・バックエンド間の分業が行いやすい
- **API の一覧性**: 全 API の定義が TypeSpec に集約されるため、一覧性が高い

### 気になる点

- **コード生成の柔軟性**: コード生成の柔軟性は言語/フレームワークとそのコードジェネレーターに依存する。エコシステムやコミュニティの充実度にも影響を受ける
